# オンボーディング再設計 設計書（レビュー9〜15対応）

クライアント第9〜15回レビューを統合した、新しいオンボーディングの設計。
「加盟者が勝手に先行できない・飛ばせない・登録しないと開始できない／手動は最小限」を徹底する。

- 参考資料：DROP AI 加盟者フロー（資金調達なし／あり）、STEP2 開業準備（3つ同時進行）
- 物販固有の項目（eBay/Payoneer 等）は引用しない。中古車FC向けに構成。

---

## 1. 全体フロー（分岐対応）

```
STEP1  契約・初期設定
        ├ 加盟契約の締結（本部承認）
        ├ アカウント発行・初回ログイン（自動）
        └ プロフィール登録（自動）
          ↓
STEP2  開業準備（★3つ同時進行）
        ├ A. 本人確認         … 必須。顔写真付き身分証を提出→本部承認
        ├ B. 古物商許可        … 取得済み/未取得で分岐。未取得でもスタート可（6ヶ月猶予）
        └ C. 資金準備          … 自己資金/資金調達で分岐
                                   ├ 自己資金：自己資金額を登録→本部確認
                                   └ 資金調達：申請→ヒアリング→書類提出→事業計画→
                                              融資申請→審査・面談→融資契約→着金確認
          ↓
STEP3  利用規約への同意（必須・ボタン押下で記録）
          ↓
STEP4  実践マニュアル（チェックボックス式・動的に項目追加/編集可）
        中古車市場の基礎 / 相場の見方 / AI壁打ちで候補整理 / 仕入れ基準 /
        出品ルール / 禁止事項・注意事項 / 理解度チェック / 修了
          ↓
利用開始条件：全STEP完了（本人確認OK + 資金準備OK + 規約同意 + マニュアル修了）
          → DROP AI（全機能）利用開始
```

> 古物商は「未取得でもスタート可・6ヶ月以内に提出」。これは STEP2 の完了をブロックしない
> 特別扱い（`optional_deadline`）とする。

## 2. 自動化／手動（本部承認）の切り分け（手動は最小限）

| タスク | 判定 | 理由 |
| --- | --- | --- |
| アカウント発行・初回ログイン | 🟢 自動 | 操作で自動 |
| プロフィール登録 | 🟢 自動 | 保存で自動 |
| 決済情報の登録 | 🟢 自動 | 登録で自動 |
| 利用規約への同意 | 🟢 自動 | 押下で自動記録 |
| 実践マニュアル各項目 | 🟢 自動 | チェックで自動 |
| 自己資金額の登録 | 🟢 自動 | 登録で自動（金額の妥当性は本部が別途確認可） |
| **本人確認書類の提出** | 🔴 本部承認 | 顔写真付き身分証の目視確認が法的に必須 |
| **古物商許可証の提出** | 🔴 本部承認 | 許可証の真正性確認（ただし猶予・任意スタート） |
| **加盟契約の締結** | 🔴 本部承認 | 契約成立の確認 |
| **資金調達の各ステップ** | 🔴 本部承認 | ヒアリング・審査・着金など人的プロセス |

→ **加盟店の操作で完結するものはすべて自動。人の目・外部プロセスが必須のものだけ本部承認。**

## 3. データモデル（DB設計）

### 3.1 onboarding_tasks 拡張（既存を活用）
```
task_kind    text  -- action | document | agreement | evidence | manual_step | approval
                    -- evidence     = ファイル提出（本人確認・古物商）
                    -- agreement    = 利用規約同意
                    -- manual_step  = 実践マニュアル項目（チェック式・動的）
completion_type text -- auto | manual（本部承認）
optional     bool  -- true=未完了でも次に進める（古物商の猶予）
group_key    text  -- 'evidence_identity' | 'funding' 等、分岐グループ
```

### 3.2 evidences（エビデンス：本人確認・古物商）
```
id, member_id, kind('identity'|'antique_license'|'other'),
doc_type('license'|'mynumber'|'passport'|'antique'),  -- 身分証の種類
storage_path, file_name, file_type, file_size,
status('pending'|'approved'|'rejected'), reviewed_by, reviewed_at, note,
created_at
```
- Storage private バケット `member-evidences`
- 加盟店がドラッグ&ドロップでアップロード → 本部が承認/却下
- 本人確認は `doc_type ∈ {license, mynumber, passport}` に制限（顔写真付き身分証）

### 3.3 funding_applications（資金調達）
```
id, member_id, method('self'|'loan'),
self_amount_yen,  -- 自己資金の場合
-- 資金調達の場合の各ステップ状態
step_status jsonb,  -- {hearing, documents, plan, apply, review, contract, deposit}
status('in_progress'|'completed'), created_at
```

### 3.4 agreements（利用規約：本部が設定・バージョン管理）
```
id, title, body(markdown), version, published, created_at   -- 本部が編集
```
### 3.5 agreement_consents（加盟店の同意記録）
```
id, member_id, agreement_id, agreed_at, ip
```

### 3.6 manual_sections（実践マニュアル：動的・編集可能）
```
id, key, title, body(markdown/rich), sort_order, published, created_at, updated_at
                                              -- 本部がローンチ後に内容を追記/編集
```
### 3.7 manual_progress（加盟店のチェック状況）
```
id, member_id, section_id, checked_at
```

## 7. 運用ルール（クライアント確定・2026-07-10）

### ① 古物商 6ヶ月猶予
- **起算日＝契約日（contract_date）**。未取得でもスタート可（スタートアップ期間を最短化）。
- **期限超過時の挙動＝一部機能制限（D）**。加えて、超過前後に**お知らせ欄／進捗ステータスに警告カラーで通知**
  「【期日が超過したので許可証をアップロードお願いします】」を目立つ形で常時表示（事前通知でチャット負荷も軽減）。
- **制限する機能**：フローチャート（オンボーディング以降の取引動線）・半自動売買フロー＝**取引そのものを停止**。
  → 目的：機能制限中に自動オーダーが走る事故を防ぐ（「制限中なのに自動発注」で管理者判断が追いつかない状態の回避）。
- **制限しない機能**：本人情報の入力、仕入れ資金など**自己管理系**、および **AI 分析**は継続利用可。

### ② 本人確認 却下時
- 却下＝**ロックせず、加盟者が再アップロードできる状態に戻す（A）**。本部の再許可は不要。

### ③ 利用規約バージョン更新
- 更新時は**既存加盟店にも再同意を求める（A）**。
- **同意ログを証拠保全目的で記録**（誰が・いつ・どのバージョンに同意したか）。規約の更新履歴はアナウンス側にも残す。

### ④ 本部サポート（古物商 等）
- 連絡手段＝**アプリ内チャット（A）**。
- 本部管理画面の**左メインメニューに「本部サポート」項目を新設**し、その中に「古物商取得サポート」等をぶら下げる（今後サポート業務が増える想定）。
- **重要（非弁類似リスク回避）**：本部が「代行」を請け負う表現は不可（無資格代行は法的にアウト・行政書士から指摘済み）。
  → **「古物商取得業者の紹介」名目**にする。UI 文言・機能名から「代行」を排し、「紹介／サポート／取次」に統一。実作業補助はグレー範囲に留める。
